# Session Error Log

## Overview

Add a per-session error journal so bugs can be researched after the fact instead of being reconstructed from memory.

**Problem.** `keys` has no logging at all — no `log`, no `slog`, no logger in `go.mod`. Being a pure TUI under an alt-screen, it cannot write to stdout/stderr either. Three classes of information are lost today:

1. **Panics in `tea.Cmd` goroutines.** Bubble Tea catches them (`recoverFromGoPanic`, tea.go:852) but calls `p.cancel()` and prints the trace *immediately* — cancellation is async, the alt-screen is still up, so the trace lands in the alt buffer and vanishes on exit. All network and subprocess code lives here, i.e. the most panic-prone paths.
2. **Silently swallowed errors.** `version.LoadCache`/`SaveCache` have four mute `return Cache{}` and three mute `return`; a corrupt or unwritable `cache.json` is completely invisible. `version.InstalledVersion` collapses every failure into `""` → `installed: not found` with no cause.
3. **Errors that reach the UI and evaporate.** `loader.SaveMeta` failures land in `m.statusMsg`, which the next keystroke erases.

**Solution.** A new dependency-free `internal/logx` package writing one plain-text file per session, created lazily on first write. A session with no errors leaves no file at all — the presence of a file is itself the signal.

**Integration.** Package-level state (like `tokenMem` in `version`), so `version`/`model`/`loader` can log without threading a logger through constructors — `New(meta)` and every `tea.Cmd` are not built for that.

## Context (from discovery)

- **Files involved:** `main.go`, `internal/version/github.go` (`LoadCache`:597, `SaveCache`:626, `doGH`:114, `classifyStatus`:148), `internal/version/detect.go` (`InstalledVersion`:18), `internal/loader/meta.go` (`SaveMeta`:82), `internal/model/commands.go` (`fetchHelpCmd`), `internal/model/model.go` (`Update`), `internal/model/render.go` (`View`).

The existing seams (`loader.testConfigDir`, `version.testCacheDir`/`testTokenDir`) are *unexported* — they only work because each package sets its own in its own tests. `logx` is different: `version`/`loader`/`model` tests must redirect its output, so its seam must be **exported** (`logx.SetDirForTesting`). This is the one place the convention cannot be copied verbatim.
- **No import cycles:** `logx` imports nothing from the project. `version` imports `loader` (`detect.go` uses `loader.Tool`), so `logx` must stay at the bottom of the graph — this is why the header is injected via `SetHeader` from `main` rather than assembled inside `logx`.
- **Bubble Tea v1.3.10 panic handling (verified in source):** `recover` in `Run` (tea.go:634-639, calling `recoverFromPanic` at tea.go:840) does `shutdown(true)` *then* prints — trace survives in scrollback. `recover` in every Cmd goroutine (call sites `recoverFromGoPanic` at tea.go:359, 511, 537, 555; the definition is at tea.go:852) prints *before* the terminal is restored (it does `p.cancel()`, which is async, then `fmt.Printf`) — trace is lost in the alt buffer. `Run` returns `fmt.Errorf("%w: %w", ErrProgramKilled, ErrProgramPanic)`.

Release binaries carry no version. Task 4 adds the variable, which both fixes the ldflags and gives the log header the single most important fact for bug research.

## Solution Overview

**Lazy per-session file.** `~/.config/keys/logs/keeptui-2026-07-17_14-32-01.log`. The file is created under the mutex on the *first* write; the header is written as its first line at that moment. No errors in a session → no file, no I/O, no cleanup pressure.

**Why per-session rather than one rotating file:** no cross-process mutex between concurrent `keys` instances, no size-based rotation, and "send me the log from that crash" becomes "send the file with that timestamp".

**Why the timestamp format is what it is:** no colons (Windows rejects them in filenames, and the release builds for windows); lexicographic order = chronological order, which is what the cleanup relies on instead of parsing dates.

**Why our `recover` sits deeper than Bubble Tea's:** `recover()` consumes the panic wherever it fires first. A `defer` above Bubble Tea's would never see the panic. So `logx.Recover` is deferred *inside* `Update`, `View` and each command, records the trace, then re-panics — after which everything proceeds as before: Bubble Tea catches it, restores the terminal, `Run` returns `ErrProgramPanic`. `tea.WithoutCatchPanics` is deliberately NOT used — terminal restoration is Bubble Tea's job and it does it correctly.

**Why `debug.Stack()` and not Bubble Tea's `debug.PrintStack()`:** inside a `defer`, the `gopanic` frame is still on the stack, so the trace contains the real panic site. Bubble Tea's print shows the recover point instead.

**Errors-only, no debug level.** Deliberately rejected — do NOT add: `Debugf`, a debug level, `KEYS_DEBUG`, any env reading, `Infof`/`Warnf`, JSON format, duplicate-collapsing, size-based rotation. Without a debug level the file appears *only* when something went wrong, so its existence is a signal rather than noise to sift.

**The logger never breaks the app.** Its own failures (file won't open, disk full) are swallowed silently — crashing or spraying the TUI because logging failed is worse than losing a log line.

## Technical Details

**API — four production functions plus one exported test seam:**

```go
func SetHeader(s string)                  // called once from main before Run
func Errorf(format string, args ...any)   // always writes
func Recover(context string)              // defer helper: recover + debug.Stack() + write + re-panic(r)
func Path() string                        // current file path, "" if never created

func SetDirForTesting(dir string) (restore func())  // redirect + reset state; other packages' tests need this
```

**State:** `sync.Mutex` guarding `file *os.File`, `path string` (what `Path()` returns — `file.Name()` is unavailable after a failed open), `header string`, and `failed bool` (a latched failed-open flag: once `OpenFile` errors we set it and never retry, so a full disk on a hot error path is not a syscall per `Errorf`). There is no separate `opened` field — `file != nil || failed` already answers "did we try". `SetDirForTesting` locks, resets all four back to zero, points the dir override at `dir`, and returns a `restore` that reverts to the real dir and re-zeros the state — so tests are order-independent (issue: sticky globals otherwise make "no file created" assertions depend on test order within the package binary).

**Line format** — plain text, one line per record, read by eye via `less`/`grep`:

```
keys v1.4.0 darwin/arm64 tools=12 token=config
2026-07-17T14:32:01.482+03:00 ERROR version.GetRepoData repo=cli/cli http=403 remaining=0: rate limited
```

Not JSON — a human reads this file, and `grep ERROR` already works.

**Header assembly** stays in `main.go` (`logx.SetHeader`) because version/tool-count/token-source belong to `main`/`loader`/`version`, and pulling them into `logx` would invert the import graph.

**Cleanup — keep the newest 20**, called synchronously from `main.go` before `p.Run()`: `os.ReadDir` over a couple dozen entries costs fractions of a millisecond, while a goroutine would race the tests. Filter by `keeptui-` prefix + `.log` suffix, sort by name, delete the tail. Foreign files in the directory are never touched.

**Logging sites (agreed list, `Errorf` only):**

| Site | What is lost today |
|---|---|
| `version.LoadCache`/`SaveCache` | corrupt/unwritable `cache.json` — totally invisible. Priority #1 |
| `version.doGH`/`classifyStatus` | network error and any non-200: HTTP code + `X-RateLimit-Remaining`, plus the request path where reachable (`classifyStatus` only receives `resp`, so repo comes from `resp.Request.URL.Path` — no signature change). Distinguishes a real rate limit from any other 403 |
| `version.InstalledVersion` | every candidate failing (`LookPath` miss / command error) → `""` with no cause. Logged **once** at the final `return ""`, never per candidate (detect.go loops `--version` then `-V`; a per-`continue` log would flag a successful `-V` fallback as an error) |
| `loader.SaveMeta` | erased from `m.statusMsg` by the next action; losing `meta.yaml` must leave a trace |
| `model.fetchHelpCmd` | `isTUITakeover` firing, or empty output behind the friendly "No --help output" |
| `main.go` | `ErrProgramPanic` from `Run` — marks "this session ended in a panic" |

**Deliberately NOT logged:** `View`/render (called dozens of times per second), keystrokes (that is a keylogger over the user's config), token contents.
